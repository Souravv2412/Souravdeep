Project 3 — Financial Data Architecture

Star Schema Data Model | Master Transaction Log (Fact) + 4 Dimension Tables | Microsoft Excel

DIMENSION TABLE 1

Vendor Directory
  • Vendor ID (PK)
  • Vendor Name
  • Contact Email
  • Payment Terms
  • Vendor Tier

DIMENSION TABLE 2

Department Cost Centers
  • Dept Code (PK)
  • Department Name
  • Department Head
  • Annual Budget ($)

FACT TABLE

Master Transaction Log
  • Invoice ID (PK)
  • Vendor ID (FK) →
  • Dept Code (FK) →
  • Status Code (FK) →
  • Invoice Date
  • Amount ($)
  • Tax ($)
  • Total ($)
  • Days Outstanding

DIMENSION TABLE 3

Tax Rate Reference
  • Jurisdiction (PK)
  • Tax Type
  • Rate (%)
  • Effective Date

DIMENSION TABLE 4

Payment Status Reference
  • Status Code (PK)
  • Description
  • Action Required
  • Escalate to Manager?
How to explain this in an interview:
"Every invoice row in the Master Transaction Log carries a Vendor ID, Dept Code, and Status Code. These are foreign keys that join to the corresponding dimension tables — so instead of storing 'Nexus Supply Co.' in 600 rows, we store VND-001 once and look it up. That's what makes the system queryable and scalable."
